### 3. Count the Number of Nodes That Are Ready to Run Normal Workloads
- Cluster: kubectl config use-context hk8s

Determine how many nodes in the cluster are `ready` to run normal workloads (i.e., workloads that do not have any special tolerations).
Output this number to the file `/var/CKA2022/count.txt`

답
@keyword taint, toleration
```
# 클러스터에서 정상적인 워크로드(즉, 특별한 tolerations이 없는 워크로드)에서 ready인 노드 수를 저장
# 힌트 worker1과 worker2 노드에는 Taint가 설정되어 있지 않습니다. 그러니 normal workload(정상적)가 실행 가능합니다.

kubectl get nodes

kubectl describe nodes hk8s-master | grep -i taint
Taints:         node-role.kubernetes.io/control-plane:NoSchedule

kubectl describe nodes hk8s-worker1 | grep -i taint
Taints:         <none>

kubectl describe nodes hk8s-worker2 | grep -i taint
Taints:         <none>

echo "2" > /var/CKA2022/count.txt
```

memo
- Taint: 노드에 적용, 자물쇠, 조건이 맞지 않는 파드의 접근 금지
- Toleration: 파드에 적용, 자물쇠 열쇠, Taint 노드에 배치

### 5. ETCD Backup & Restore
- console에서 k8s master의 etcd backup & restore
- 작업 클러스터 : k8s

First, create a snapshot of the existing etcd instance running at `https://<k8s-master's IP 10.0.2.10>:2379`, saving the snapshot to `/data/etcd-snapshot.db`.
Next, restore an existing, previous snapshot located at `/data/etcd-snapshot-previous.db`.
The following TLS certificates/key are supplied for connecting to the server with etcdctl:
- CA certificate: `/data/cka/ca.crt`
- Client certificate: `/data/cka/server.crt`
- Client key: `/data/cka/server.key`

답
@keyword etcd backup
```
### etcd snapshot
# console

etcd --version

ETCDCTL_API=3 etcdctl --endpoints=https://127.0.0.1:2379 \
  --cacert=<trusted-ca-file> --cert=<cert-file> --key=<key-file> \
  snapshot save <backup-file-location>

ETCDCTL_API=3 etcdctl --endpoints=https://10.0.2.10:2379 \
  --cacert=/data/cka/ca.crt --cert=`/data/cka/server.crt --key=/data/cka/server.key \
  snapshot save /data/etcd-snapshot.db

### restore
ssh k8s-master

su -i (root 권한 얻기)
sudo -i

ls /data

etcdutl --data-dir <data-dir-location> snapshot restore snapshot.db

etcdutl --data-dir /var/lib/etcd-new snapshot restore /data/etcd-snapshot-previous.db

vim /etc/kubernetes/manifests/etcd.yaml (static pod 자동으로 반영, 2~3분 걸림)

(etcd를 바라보는 디렉토리 수정)

AS-IS
- hostPath:
    path: /var/lib/etcd

TO-BE
- hostPath:
    path: /var/lib/etcd-new
```

memo
- etcd: cluster의 모든 information를 갖고 있다. 메모리 공간. (`/var/lib/etcd/*`)

### 6. Cluster Upgrade - only Master
- 작업 클러스터 : kubectl config use-context k8s

upgrade system : k8s-master
Given an existing Kubernetes cluster running version `1.30.0`, upgrade all of the Kubernetes control plane and node components on the master node only to version `1.30.6`.
Be sure to drain the master node before upgrading it and uncordon it after the upgrade.

답
@keyword kubeadm upgrade
```
kubectl get nodes

ssh k8s-master

sudo -i

kubectl get nodes

apt-get update
apt-cache madison kubeadm
# apt-cache madison kubectl
# apt-cache madison kubelet

sudo apt-get update
sudo apt-get install -y kubeadm='1.30.6-1.1'
sudo apt-mark hold kubeadm

sudo kubeadm upgrade plan (검증)

kubeadm version

sudo kubeadm upgrade apply v1.35.x -y

kubectl drain k8s-master --ignore-daemonsets

sudo apt-mark unhold kubelet kubectl
sudo apt-get update && sudo apt-get install -y kubelet='1.35.x-*' kubectl='1.35.x-*'
sudo apt-mark hold kubelet kubectl

sudo systemctl daemon-reload
sudo systemctl restart kubelet

kubectl uncordon k8s-master
```

### 7. Authentication and Authorization
- Cluster : k8s

Context

You have been asked to create a new ClusterRole for a deployment pipeline and bind it to a specific ServiceAccount scoped to a specific namespace.

Task:
- Create a new `ClusterRole` named deployment-clusterrole , which only allows to create the following resource types: `Deployment` `StatefulSet` `DaemonSet`
- Create a new `ServiceAccount` named cicd-token in the existing namespace `app-team1`.
- Bind the new `ClusterRole` deployment-clusterrole to the new `ServiceAccount` `cicd-token`, limited to the namespace `app-team1`.

답
@keyword ServiceAccount RBAC
```
kubectl create serviceaccount cicd-token -n app-team1
kubectl get serviceaccount -n app-team1

kubectl create clusterrole deployment-clusterrole --verb=create --resource=deployment,statefulset,daemonset

kubectl create clusterrolebiding test-binding --clusterrole=deployment-clusterrole --serviceaccount=cicd-token
```

memo
- Role을 ServiceAccount에 연결을 RoleBinding, ClusterRoleBinding한다
- RBAC(Role-Based Access Control) -> Role(특정 네임스페이스), ClusterRole(클러스터 전체 범위)
  Role을 사용해야 할 때
  파드(Pod), 서비스(Service), 컨피그맵(ConfigMap) 등 특정 네임스페이스 격리 환경 안에서만 권한을 제어할 때.
  ClusterRole을 사용해야 할 때
  노드(Node), 퍼시스턴트 볼륨(PV) 등 네임스페이스에 속하지 않는 글로벌 리소스를 제어할 때.
  /healthz 같은 비-리소스(Non-resource) 엔드포인트에 접근할 때.
  클러스터 내 모든 네임스페이스에 걸쳐 특정 리소스를 조회하거나 관리할 때 (예: kubectl get pods -A).

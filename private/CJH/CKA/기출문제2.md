### 02
Update the existing deployment wordpress, adding a sidecar container named sidecar using the busybox:stable image to the existing pod.

The new sidecar container has to run the following command: "/bin/sb -c "tail -f /var/log/wordpress.log" use a volume mounted at /var/log to make the log file wordpress.log available to the co-located container

답
```
kubectl get deploy
kubectl edit deploy

volumes:
    - name: log
      emptyDir: {}

volumeMounts:
    - name: log
      mouthPath: /var/log

- name: sidecar
  image: busybox:stable
  command: ["/bin/sb -c "tail -f /var/log/wordpress.log"]
  volumeMounts:
    - name: sidecar
      mountPath: /var/log

kubectl get deploy -w
kubectl describe deploy
```

### 03

You have an existing web application deployed in a Kubernetes cluster using an Ingress resource named web. You must migrate the existing Ingress configuration to the new Kubernetes Gateway API, maintaining the existing HTTPS access configuration.

Tasks:

Create a Gateway resource named web-gateway with hostname gateway.web.k8s.local that maintains the existing TLS and listener configuration from the existing Ingress resource named web.

Create an HIIPRoute resource named web-route with hostname gateway.web.k8s.local that maintains the existing routing rules from the current Ingress resource named web.

Note: A GatewayClass named nginx-class is already installed in the cluster.

답
```
kubectl get secret
kubectl get ingress
kubectl describe ingress
kubectl get service

vi gateway.yaml

apiVersion: gateway.networking.k8s.io/v1
kind: Gateway
metadata:
  name: web-gateway
spec:
  gatewayClassName: nginx-class
  listeners:
  - name: https
    protocol: HTTPS
    port: 443
    hostname: "gateway.web.k8s.local"
    tls:
      mode: terminate
      certificateRefs:
        - kind: Secret
        - name: web-tls

kubectl describe ingress

kubectl apply -f gateway.yaml

apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
  name: web-route
spec:
  parentRefs:
  - name: web-gateway
  hostnames:
  - "gateway.web.k8s.local"
  rules:
  - matches:
    - path:
        type: PathPrefix
        value: /login
    backendRefs:
    - name: web-svc
      port: 80

kubectl get service
kubectl apply -f httproute.yaml
kubectl get gateway,httproute
kubectl describe gateway,httproute

kubectl get secret web-tls
```

### 05
Create a new StorageClass named local-kiddie with the provisioner rancher.io/local-path.

Set the volumeBindingMode to WaitForFirstConsumer

Configure the StorageClass as te default StorageClass.

Do not modify any existing Depoyments or PersistentVolumeClaims

답
```
kubectl get sc

vi sc.yaml

apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: local-kiddie
  annotations:
    storageclass.kubernetes.io/is-default-class: "false"
provisioner: rancher.io/local-path
volumeBindingMode: WaitForFirstConsumer

kubectl apply -f sc.yaml

kubectl patch sc local-kiddie -p '{"metadata:{"annotation":{"storageclass.kubernetes.io/is-default-class":"true"}}}'

kubectl patch sc local-path -p '{"metadata:{"annotation":{"storageclass.kubernetes.io/is-default-class":"false"}}}'
```

### 06

in a namespace called priority.

The cluster already has at least one user-defined Priority Class

Perform the following tasks:

1. Create a new Priority Class named high-priority for user workloads. The value of this Priority Class
should be exactly one less than the highest existing user-defined Priority Class value.

2. Patch the existing Deployment busybox-logger in the priority namespace to use the newly created
high-priority Priority Class

답
```
kubectl get pc

kubectl create pc high-priority --value=999 --description="high-priority-workloads"

kubectl get pc

kugectl get deploy -n priority

kubectl get deploy busybox-logger -n priority -o yaml

kubectl patch deploy -n priority busybox-logger -p '{"spec":{"template":{"spec":{"priorityClassName":"high-priority"}}}}'

kubectl get deploy busybox-logger -n priority -o yaml
```
